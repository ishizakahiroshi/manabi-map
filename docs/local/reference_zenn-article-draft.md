## 作ったもの

**Manabi Map（まなびマップ）** — 住所を入れると通える高校が地図にピンで表示され、気になる学校をお気に入り保存して、見学メモや通学メモを家族で共有できるサービスです。

https://manabi-map.app

- 関東 7 都県・全日制など 1,362 校
- 無料・アカウントなしでも地図検索は使える（保存機能は Google / LINE ログイン or 匿名認証）
- コードは AGPL-3.0 で公開（データは CC BY-SA 4.0）

（ここにヒーロー画像: ピンが立った地図のスクショ）

## なぜ作ったか

高校選びの情報は「偏差値順の一覧サイト」に偏っていて、親子が最初に知りたい「うちから現実的に通えるのはどの学校か」に答えてくれる地図がありませんでした。偏差値ランキングではなく、**進路の家族会議を支えるノート**を作りたかった、というのが出発点です。

## 技術スタック

- フロント: React 19 + TypeScript (strict) + Vite + Tailwind CSS v4
- 地図: Leaflet + OpenStreetMap（ラッパーを使わず素の Leaflet を useRef/useEffect で制御）
- バックエンド: Supabase（PostgreSQL + Auth + RLS）。専用 API サーバーなし、フロントから直接
- 認証: Google OAuth + LINE ログイン（Custom Provider）+ 匿名認証
- ホスティング: Cloudflare Pages（main への push = 本番デプロイ）
- 費用: ほぼドメイン代のみ（年間数千円）

## ハマりどころ 0: OAuth コールバックのエラーはクエリで返ってくる（黙って失敗させない）

Supabase 経由の OAuth（LINE / Google）は、失敗時にコールバック URL の `error` / `error_code` / `error_description` クエリでエラーを返してきます。ここを読まずにセッションだけ見ると「なぜか未ログイン画面に戻る」だけの謎挙動になり、v0.2.3 リリース直後にユーザーから「LINE ログインが壊れた」と報告を受けました。

v0.2.4 で `AuthCallbackPage.tsx` にクエリ検知を入れ、特に `identity_already_exists`（匿名アカウントに OAuth を linkIdentity しようとしたが、その OAuth ID は別ユーザーで既に登録済み）は「一度ログアウトして直接ログインしてください」の具体的な案内文言に振り分けています。汎用エラーとタイムアウトは別文言。i18n（ja/en）化済み。

一般化すると、OAuth 実装で「セッションだけ見る」設計はダメで、URL のエラークエリを最初に見ないと詰まった時に何も分からなくなります。

## ハマりどころ 1: LINE ログイン × Supabase は素直に OIDC で繋ぐと必ず失敗する

この記事で一番書きたかった部分です。Supabase の Custom OIDC Provider に LINE を素直に登録すると、**LINE のウェブログインが返す ID トークンが HS256 署名**（チャネルシークレットで署名）なのに対し、**Supabase 側は ES256/RS256 の JWKS 検証を要求**するため、トークン検証で必ずコケます。

回避手順（結論だけ）:

1. まず scope に openid を**入れずに** Custom Provider を作成する
2. Provider を非 OIDC タイプとして構成する
3. userinfo エンドポイントに `https://api.line.me/oauth2/v2.1/userinfo` を指定し、JWKS は空欄にする
4. 最後に scope へ openid を追加する

この順番でないと OIDC 扱いに倒れて HS256 問題を踏みます。「LINE Supabase 連携」で検索して出てくる情報だけだと詰まりやすいポイントでした。

## データの話: 偏差値は商用サイトから「転載しない」と決めた

学校データで一番センシティブなのが偏差値です。商用の偏差値サイトからのスクレイピング・転載は行わず、公的資料（入試結果・倍率等）に基づく独自推計として `source_type='manabi_estimate'` を付けて管理しています。個人開発だと「よそから数字を持ってくる」誘惑が強い領域ですが、ここを雑にするとサービスの信頼が終わるので、データ出自をスキーマレベルで持つ設計にしました。

（→ スキーマの断片と、推計メソッドをバージョン管理している話）

## 運用の話: 広告は「進路に関係あるもの」しか載せない縛り

中高生と保護者が使うサービスなので、AdSense のような無差別配信は入れない・モーダルや追従バナーはやらない、という広告ポリシーをリポジトリの公開ドキュメントに明文化しています。収益より信頼を優先する設計判断も、個人開発だからこそ強制力を持たせたい部分でした。

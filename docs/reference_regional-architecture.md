# 地域設計ビジョン — 「全国展開できる器 × その地方でベスト」

> この文書は Manabi Map の **地域（リージョン）設計の背骨** です。開発が進んでも判断がぶれないよう、
> プロダクトの地域ビジョンと、その実装方針を 1 か所に固定します。

## 0. この文書の読み方（読者別）

Manabi Map（まなびマップ）は「親子で使う、学校選びの地図ノート」。この文書は 2 種類の読者を想定しています。

- **プロダクトオーナー（非エンジニア）向け** … 第 1〜4 章。なぜ「地方単位」で作るのか、
  「関東でベスト」とは何を意味するのかを、専門用語なしで説明します。まずここだけ読めば方向性は掴めます。
- **実装するエンジニア / AI エンジニア向け** … 第 5〜8 章。同じ思想を「コードでどう表現するか」に翻訳します。
  第 4 章の関東の具体値（対象県・地図範囲・郵便番号）と、第 6 章の `Region` 抽象がセットで実装の正典です。

専門用語には初出時に *（注: …）* を付けます。まとまった用語集は第 9 章にあります。

---

## 1. いちばん大事な一文

**Manabi Map は最終的に全国展開する。しかし「全国を薄く 1 枚で覆う巨大サイト」は作らない。
地方（リージョン）を 1 つずつ、その地方に住む親子にとって完璧な状態に仕上げてから、次の地方を足していく。**

いまアクティブな地方は **関東 1 都 6 県**（東京・神奈川・埼玉・千葉・茨城・栃木・群馬）。
この文書を書いている時点で、私たちが全力で「ベスト」にするのは、この 7 都県の中の体験です。

---

## 2. プロダクトのビジョン — なぜ「地方単位」なのか

### 2.1 価値の源泉は「深さ」であって「広さ」ではない

Manabi Map の価値は「日本中の高校を並べられること」ではありません。
**その地方に住む親子が、自分たちの通える範囲の学校選びを、深く・安心して進められること**です。

- 群馬に住む中学生にとって必要なのは、北海道や九州の高校ではなく、通える群馬・近県の高校の情報
- 東京に住む親子にとっては、都内の私立高校が主役になる場面が多い

だから、機能の作り込みは「全国を広く」ではなく「その地方を深く」の方向に投資します。
広さ（対応エリアの数）は、深さを 1 つずつ完成させた結果として、後から増えていきます。

### 2.2 それでも「全国展開」は捨てない

一方で、最終ゴールは全国です。ここで陥ってはいけないのが、
**「関東専用アプリ」を作り込んでしまい、次の地方を足すときに作り直しになる**こと。

そこで本プロダクトは、次章の「地方（リージョン）」という考え方で、
**全国に広げられる汎用の“器”**と、**その地方に最適化された“中身”**を分けて設計します。

---

## 3. 「地方（リージョン）」という考え方 — 器と中身

### 3.1 器と中身のたとえ

お弁当箱（器）とおかず（中身）で考えると分かりやすいです。

- **器（全国共通・変えない）** … 「地方には “対象の都道府県” と “地図で見せる範囲” と “郵便番号の扱い” がある」
  という枠組みそのもの。この枠組みは関東でも関西でも東北でも同じ形をしている
- **中身（地方ごとに入れ替える）** … 関東という器に入れる具体的な値
  （対象は 1 都 6 県、地図の初期範囲はこのあたり、郵便番号はこの範囲、など）

新しい地方を足すときは、**器はそのまま、中身だけ入れ替える**。
これで「全国に広げられる（＝器が共通）」と「その地方でベスト（＝中身が最適化されている）」が両立します。

```
        ┌─────────────── 器（全国共通の枠組み） ───────────────┐
        │  Region = { 対象の都道府県, 地図の範囲, 地図の初期表示,  │
        │             郵便番号の扱い, 表示名 }                    │
        └───────────────────────────────────────────────────────┘
                 ▲                 ▲                 ▲
                 │ 中身を入替       │                 │
         ┌───────┴──────┐  ┌───────┴──────┐  ┌───────┴──────┐
         │ 関東（今ここ）│  │ 関西（将来）  │  │ 東北（将来）  │
         │ 1 都 6 県     │  │ 2 府 4 県     │  │ 6 県          │
         └──────────────┘  └──────────────┘  └──────────────┘
```

### 3.2 いま「アクティブな地方」は 1 つだけ

同時にいくつもの地方を中途半端に開くことはしません。
**「いま作り込んでいる地方＝アクティブなリージョン」を 1 つに定め**、そこをベストにする。
関東が完成したら、次の地方をアクティブに加えていきます。

---

## 4. いまアクティブな地方 = 関東 1 都 6 県

### 4.1 対象の都道府県

東京都・神奈川県・埼玉県・千葉県・茨城県・栃木県・群馬県 の 7 都県。
このどれかに属する場所・学校が「圏内」で、それ以外（例: 北海道・長野・福島）は「圏外」です。

### 4.2 地図で見せる範囲（bbox）

*（注: bbox = bounding box。地図上の長方形の範囲を「西端・南端・東端・北端」の 4 つの数字で表したもの）*

関東 1 都 6 県は、おおむね次の長方形に収まります（島嶼部は当面対象外）。

- 西端 経度 約 138.4 / 東端 経度 約 140.9
- 南端 緯度 約 34.9 / 北端 緯度 約 37.16

この長方形を「関東の範囲」として、検索の優先や地図の初期表示に使います。

### 4.3 郵便番号の範囲（きれいな事実）

関東 1 都 6 県は、郵便番号の上 3 桁が **ほぼ 100〜379 に連続で収まります**。

- 東京 100〜208 / 神奈川 210〜259 / 千葉 260〜299 / 茨城 300〜319 /
  栃木 320〜329 / 埼玉 330〜369 / 群馬 370〜379

この範囲を「関東の郵便番号」として扱えます（全国の郵便番号表を持たなくても、関東かどうかを判定できる）。

## 5. 検索体験の 5 原則（ここが今回の修正の核心）

「住所を入れると通える高校が地図に出る」— この入口が Manabi Map の第一印象です。
関東版では、検索を「全国から関東を探す」のではなく、**「最初から関東の中だけを見る」**体験にします。

### 原則 1. 圏外の候補は出さない

アクティブな地方（いまは関東）の外にある候補は、検索結果に出しません。

- 悪い例（現状）: 「沼田」と入れると **北海道雨竜郡沼田町** が先頭に出る
- あるべき姿: 「沼田」と入れたら **群馬県沼田市** が出る。北海道は出さない

### 原則 2. 住所・地名で確実に着地する

「前橋市」「さいたま市」「千代田区」などの住所・地名は、関東の中で正確な地点に着地させます。

### 原則 3. 郵便番号で確実に着地する

関東 1 都 6 県の郵便番号（100〜379 番台）は、上 3 桁でも 7 桁でも地点に着地させます。

- 現状の問題: 郵便番号は**群馬（370〜379）だけ**を手書きの表で持っており、
  東京・埼玉など他県の郵便番号は反映されなかった → これを関東全域に拡張する

### 原則 4. 駅名で着地する

「高崎駅」「大宮駅」などの駅名でも中心地点に着地させます。

### 原則 5. お店・ランドマーク混じりの入力でも、地名を拾って着地する

親は Google 検索の感覚で「沼田 ヤマダ電機 群馬」のように入れることがあります。
住所検索の仕組みは店名そのものは苦手ですが、**入力の中から地名（沼田・群馬）を拾って**、
群馬県沼田市に着地させます。店の正確な位置ではなく、その周辺（中心地点）に寄れれば十分です。

*（注: Manabi Map はユーザーが入れた地点を「自宅住所」ではなく「中心地点」として扱います。
　ピンポイントの住所は不要で、通学の目安になる周辺に寄れれば目的を果たせます）*

## 6. 実装アーキテクチャ（エンジニア向け）

ここからは第 2〜5 章の思想を、コードでどう表現するかの仕様です。

### 6.1 `Region` 抽象（器）

地方 1 つを表す設定オブジェクトを定義し、`ACTIVE_REGION` として 1 つだけ選ぶ。
関東という文字列・数値をコードのあちこちにベタ書きしない。すべてここを参照する。

```ts
export interface Region {
  /** 内部 ID（例: 'kanto'） */
  id: string
  /** 表示名（i18n キー or ja/en） */
  labelJa: string
  labelEn: string
  /** 対象の都道府県名（「東京都」「群馬県」…完全一致で判定に使う） */
  prefectures: string[]
  /** 地理範囲 bbox: [west, south, east, north]（経度・緯度） */
  bbox: [number, number, number, number]
  /** 地図の初期表示（このリージョンを開いたときの中心とズーム） */
  mapCenter: { lat: number; lng: number }
  mapZoom: number
  /** 郵便番号 上 3 桁の範囲（[min, max] の配列。関東は [[100, 379]]） */
  postalPrefixRanges: [number, number][]
}
```

関東の中身（値）:

```ts
export const KANTO: Region = {
  id: 'kanto',
  labelJa: '関東 1 都 6 県',
  labelEn: 'Greater Tokyo (Kanto)',
  prefectures: ['東京都', '神奈川県', '埼玉県', '千葉県', '茨城県', '栃木県', '群馬県'],
  bbox: [138.4, 34.9, 140.9, 37.16],
  mapCenter: { lat: 36.0, lng: 139.4 }, // ※実機で調整する
  mapZoom: 9,                            // ※実機で調整する
  postalPrefixRanges: [[100, 379]],
}

/** いまアクティブな地方。全国展開時はここを差し替える（or env で選ぶ）。 */
export const ACTIVE_REGION: Region = KANTO
```

### 6.2 検索（geocoding）が `Region` を参照する

`web/src/lib/geo.ts` の `geocodeSearch()` を、`ACTIVE_REGION` を軸に組み替える。
（実測で確認済みの前提: **GSI は郵便番号・店名を引けない / Nominatim は関東 viewbox で圏内優先＆郵便番号解決が可能**）

処理方針（第 5 章の 5 原則に対応）:

1. **郵便番号**（`postalPrefixRanges` に入る数字入力）→ Nominatim に `viewbox = bbox` + `bounded=1` で問い合わせ、
   圏内の地点を返す。**群馬ハードコードの `POSTAL_MAP` は廃止**（原則 3）
2. **駅名**（「駅」「Station」を含む）→ Nominatim（`viewbox = bbox`、圏内優先）（原則 4）
3. **通常の住所・地名** → まず GSI で引き、`prefectures` と `bbox` で圏内フィルタ＆圏内を上位ソート。
   圏内候補が無ければ Nominatim（`viewbox = bbox` + `bounded=1`）へフォールバック（原則 1・2）
4. **店名混じりの複合入力**（スペース区切り 2 語以上で 3 の結果が空）→ 入力をトークン分解し、
   地名らしいトークンで Nominatim（圏内 `bounded=1`）を引き直して着地（原則 5）

`bbox` / `prefectures` を使った圏内判定と圏内優先ソートは共通ヘルパにまとめ、
「関東」を直接書かない（`ACTIVE_REGION` を差し替えれば別地方でそのまま効く）。

### 6.3 地図の初期表示が `Region` を参照する

地図ページの初期中心・ズームは `ACTIVE_REGION.mapCenter` / `mapZoom` を使う。
群馬中心の固定値をベタ書きしない。

### 6.4 UI 文言

「お店の名前で OK」という過大約束は実態に合わせて調整する
（住所・郵便番号・駅名を主に案内し、店名混じりでも地名で拾える旨は補助的に示す）。
プレースホルダの例示も関東 1 都 6 県を代表する地名に更新する。

## 7. 全国展開の手順（将来）

新しい地方を足すときにやることは、原則として **`Region` 定義を 1 つ追加して `ACTIVE_REGION` に加える**だけ。

1. 例: `KANSAI: Region = { prefectures: ['大阪府', '京都府', …], bbox: […], … }` を定義
2. 学校データ・偏差値データをその地方ぶん投入（データ投入の手順は別途 skill / plan に従う）
3. `ACTIVE_REGION`（または対応リージョンの選択ロジック）に追加
4. 検索・地図・郵便番号は `Region` を参照しているので、コード本体の改修は最小

「その地方でベスト」を再現するために、各リージョンの中身（bbox・初期表示・対象県）は
実機で確認しながらチューニングする。器は使い回し、中身だけ地方に合わせて磨く。

## 8. やらないこと（境界）

- **全国横断検索はしない。** アクティブな地方の外を検索結果に混ぜない（原則 1）
- **全国の郵便番号表を持たない。** リージョンの `postalPrefixRanges` と圏内ジオコーディングで解決する
- **店名そのものの正確な位置特定はしない。** 中心地点として周辺に寄れれば十分（原則 5）
- **地方をいくつも同時に中途半端に開かない。** アクティブな地方を 1 つずつ完成させる（第 3.2 節）
- 巨大偏差値サイト化・ランキング化はしない（プロダクト全体の方針。`CLAUDE.md` 参照）

## 9. 用語集（非エンジニア向け）

- **リージョン（地方）** … Manabi Map が対応する地方の単位。いまは「関東 1 都 6 県」。全国展開は
  このリージョンを 1 つずつ増やしていく形で進む
- **ジオコーダ / ジオコーディング** … 「前橋市」「371-0022」などの文字を、地図上の座標（緯度・経度）に
  変換する仕組み。本アプリは国土地理院（GSI）と OpenStreetMap（Nominatim）の 2 つを使い分ける
- **GSI（国土地理院）** … 日本の住所に強い公式の地図サービス。ただし郵便番号と店名は引けない
- **Nominatim（OpenStreetMap）** … 郵便番号や地理範囲の指定（viewbox）に対応するジオコーダ
- **bbox（バウンディングボックス）** … 地図上の長方形の範囲。西・南・東・北の 4 数字で表す
- **viewbox / bounded** … Nominatim に「この範囲を優先して（bounded=1 なら範囲外を除外して）探して」と
  指定する仕組み。これで「関東の中だけ」を実現する
- **中心地点** … ユーザーが検索で置く基準の地点。自宅住所そのものではなく、通学範囲を測る中心として扱う
